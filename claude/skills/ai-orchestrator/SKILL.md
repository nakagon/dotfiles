# AI Orchestrator

Claude をコンダクター（指揮者）として、Codex (GPT) と Gemini を
コンサルタントとして活用し、複数AIモデルの視点を統合した開発・レビューを行う。

## 基本原則

1. **Claude がリード** — プロジェクトの全コンテキストを持つ自分が指揮する。他モデルは専門コンサルタント
2. **結果は統合して提示** — 各モデルの生出力をそのまま見せない。要点を抽出し、一貫した回答を構築する
3. **コスト意識** — 外部モデル呼び出しはトークンコストがかかる。明確な目的があるときだけ使う
4. **検証してから採用** — 外部モデルの回答を盲目的に信頼しない。自分の知識・コンテキストで検証する

## 利用可能なモデル

### Codex (GPT)

**ツール**: `mcp__codex__codex`
**得意分野**: アーキテクチャ設計、コードレビュー、セキュリティ分析、計画レビュー

5種の Expert が利用可能:

| Expert | 得意分野 | 使うとき |
|--------|---------|---------|
| Architect | システム設計、トレードオフ分析 | 設計判断、DB設計、API設計、2回以上修正失敗時 |
| Plan Reviewer | 計画の検証 | 実装前の計画レビュー |
| Scope Analyst | 要件分析 | 曖昧な要件の明確化、スコープ分析 |
| Code Reviewer | コード品質、バグ発見 | コードレビュー、PR前の自己レビュー |
| Security Analyst | 脆弱性、脅威モデリング | 認証/認可変更、機密データ処理、API追加 |

**委任方式**: `.claude/rules/delegator/delegation-format.md` の7セクション形式に準拠。
委任前に Expert のプロンプトファイルを読み込むこと（`orchestration.md` Step 2 参照）。

### Gemini

**チャット**: `mcp__gemini-cli__geminiChat` — コードレビュー、設計相談、代替案提案
**検索**: `mcp__gemini-cli__googleSearch` — 最新ドキュメント、既知の問題、ベストプラクティス

Gemini の強みは **最新情報へのアクセス** と **異なるモデルファミリーの視点**。
Claude や GPT と異なる学習データ・推論パターンを持つため、死角の補完に有効。

モデル選択: `gemini-3-pro-preview`（デフォルト、深い分析向け）/ `gemini-3-flash-preview`（高速応答向け）

### Claude (自身)

**役割**: コンダクター、実装者、統合者
**強み**: 会話の全コンテキスト保持、ファイル操作、全ツール連携

## オーケストレーションパターン

### パターン 1: クロスモデルレビュー

**目的**: 複数モデルにコードや設計をレビューさせ、死角を減らす
**使うとき**: PR前の重要なコードレビュー、設計レビュー

**手順**:
1. レビュー対象（コード、diff、設計ドキュメント）を準備
2. **並行して** Codex と Gemini にレビューを依頼
3. 両方の指摘を統合し、重要度順に整理
4. 統合レビュー結果をユーザーに提示

**Codex**: Code Reviewer Expert に7セクション形式で委任
**Gemini**: 以下のテンプレートで依頼

```
mcp__gemini-cli__geminiChat({
  prompt: `以下のコードをレビューしてください。

重点確認: バグ/論理エラー、パフォーマンス、セキュリティ、保守性

コンテキスト:
- [プロジェクトの背景を簡潔に]
- [技術スタック: Python/FastAPI, TypeScript/Next.js 等]

コード:
\`\`\`
[対象コード]
\`\`\`

指摘は重要度（Critical/Warning/Info）で分類し、修正案を添えてください。`,
  model: "gemini-3-pro-preview"
})
```

**統合の形式**:
- **Critical** (両モデル一致、または明らかなバグ) — 必ず修正
- **Warning** (片方が指摘) — 修正を推奨
- **Info** (改善提案) — 検討事項として提示

---

### パターン 2: リサーチ + 判断

**目的**: Gemini で最新情報を調査し、Codex の専門知識で意思決定する
**使うとき**: ライブラリ選定、技術選択、未知のエラー調査

**手順**:
1. Gemini Search でトピックを検索
2. Gemini Chat で検索結果を深堀り
3. Codex Architect にトレードオフ分析を依頼（検索結果を含める）
4. Claude が判断を統合

```
# Step 1: 検索
mcp__gemini-cli__googleSearch({
  query: "[具体的な検索クエリ]",
  raw: true
})

# Step 2: 深堀り
mcp__gemini-cli__geminiChat({
  prompt: "[検索結果を踏まえた詳細質問]",
  model: "gemini-3-pro-preview"
})

# Step 3: Codex Architect に委任（7セクション形式）
# 検索結果・Geminiの分析を CONTEXT セクションに含める
```

---

### パターン 3: デバッグエスカレーション

**目的**: 自力解決できないバグに対し、外部モデルの新鮮な視点を得る
**使うとき**: 同じ問題で2回以上の修正試行が失敗

**手順**:
1. 問題の状況をまとめる（エラー、試した修正、結果）
2. **並行して**:
   - Gemini で同様の問題の解決事例を検索
   - Codex Architect に新鮮な視点での分析を依頼
3. 得られた知見を統合して新たな修正方針を立てる

Codex への委任では、PREVIOUS ATTEMPT セクションを含めること:
```
PREVIOUS ATTEMPT:
- What was done: [試した修正の要約]
- Error encountered: [正確なエラーメッセージ]
- Files modified: [変更したファイル]
```

---

### パターン 4: セキュリティ監査

**目的**: セキュリティに関わる変更を多角的にチェック
**使うとき**: 認証/認可の変更、機密データ処理、新規APIエンドポイント追加

**手順**:
1. 対象コードを特定
2. Codex Security Analyst に脆弱性分析を依頼
3. Gemini で最新の CVE・脆弱性情報を検索
4. 結果を統合し、リスク評価と修正計画を提示

```
# Gemini で最新脆弱性を検索
mcp__gemini-cli__googleSearch({
  query: "[ライブラリ名] CVE vulnerability 2025 2026",
  raw: true
})
```

---

### パターン 5: 設計判断 (ADR)

**目的**: 重要なアーキテクチャ決定について複数視点で検証
**使うとき**: 新機能の設計、大規模リファクタリング、技術的意思決定

**手順**:
1. Claude が選択肢と制約条件を整理
2. Codex Architect にトレードオフ分析を依頼
3. Gemini で業界のベストプラクティスを調査
4. Claude が ADR (Architecture Decision Record) を作成

## パターン選択ガイド

| ユーザーの意図 | パターン | 使うモデル |
|-------------|---------|-----------|
| 「レビューして」「コード見て」 | 1. クロスレビュー | Codex + Gemini |
| 「どれがいい？」「比較して」 | 2. リサーチ+判断 | Gemini → Codex |
| 「なぜ動かない」「バグが直らない」 | 3. デバッグ | Gemini + Codex |
| 「セキュリティ大丈夫？」 | 4. セキュリティ | Codex → Gemini |
| 「設計どうする」「アーキテクチャ」 | 5. ADR | Codex + Gemini |
| 「GPTに聞いて」（単一指定） | 単一委任 | Codex のみ |
| 「Geminiで調べて」（単一指定） | 単一委任 | Gemini のみ |

**単一モデル指定**の場合は、指定されたモデルだけに委任する。
複数モデルを使うかは**ユーザーの意図と問題の複雑さ**で判断する。

## Codex への委任

Codex への委任は `.claude/rules/delegator/` の既存ルールに完全準拠する:

1. **Expert 選択**: `triggers.md` のシグナルマッチングに従う
2. **プロンプト読み込み**: `orchestration.md` Step 2 に従い Expert のプロンプトファイルを読む
3. **7セクション形式**: `delegation-format.md` に従いプロンプトを構築
4. **モード選択**: 分析なら Advisory (`read-only`)、実装なら Implementation (`workspace-write`)

**重要**: Codex は **ステートレス**。毎回の呼び出しに必要なコンテキストを全て含めること。

## 統合レポートの形式

複数モデルの回答を統合する際、以下の形式で報告する:

```markdown
## [タスク名] オーケストレーション結果

### 概要
[1-2文で結論]

### Codex (GPT) の見解
[要約 — 主要な指摘と推奨]

### Gemini の見解
[要約 — 主要な指摘と推奨]

### 統合分析
[一致点、相違点、Claude自身の分析]

### 推奨アクション
1. [最も重要なアクション]
2. [次に重要なアクション]
```

**一致した場合**: 信頼度が高い。「両モデルが一致して推奨」と明記。
**相違した場合**: 各モデルの根拠を整理し、トレードオフを提示。一方的に選ばない。

## フォールバック戦略

| 状況 | 対応 |
|------|------|
| Codex 利用不可 | Gemini + Claude で対応。ユーザーに「Codex 不可」と通知 |
| Gemini 利用不可 | Codex + Claude で対応。Web検索は `WebSearch` で代替 |
| 両方利用不可 | Claude 単独で対応。「外部モデル連携なし」と通知 |
| 片方がエラー | もう一方の結果を使用。エラー内容を報告 |

ツールが利用可能かは、呼び出し前に確認する。
利用不可の場合は**代替手段を提示**し、ユーザーに判断を委ねる。

---

## 注意事項

- **機密情報を送信しない** — APIキー、パスワード、個人情報を外部モデルに含めない
- **コンテキストは明示的に** — 外部モデルは会話の流れを知らない。必要情報を全て含める
- **検証してから採用** — 外部モデルの回答は必ず自分で検証する
- **ユーザーに事前通知** — 外部モデルに委任する前に「Codex/Gemini に聞きます」と伝える
- **日本語で報告** — 統合結果はユーザーとの会話言語（日本語）で報告する
